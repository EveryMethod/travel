# 逐段地图通勤估算设计

## 背景

当前旅游规划核心流程已经能收集目的地、天气、路线和价格上下文，生成按日行程，并做基础预算、偏好和节奏检查。下一步迭代聚焦一个最容易提升结果可信度的问题：同一天相邻活动之间是否真的走得动。

## 目标

在不增加新服务和复杂前端的前提下，为生成后的行程增加“逐段地图通勤估算”。系统检查每天相邻活动的地点距离和预计耗时，发现明显跨区或通勤过长时，把风险提醒追加到现有 `tips` 中。

示例提醒：

> 第 1 天 故宫 → 环球影城 通勤约 80 分钟，建议拆到不同日期或删除其中一个。

## 非目标

- 不做全局路线优化。
- 不重排整天行程。
- 不新增数据库字段。
- 不新增前端复杂 UI。
- 不保证地图数据实时准确；只作为出发前核对提醒。

## 流程设计

现有流程保持不变，只扩展生成后的 `_review_plan`：

1. 遍历 `plan.days`。
2. 每天最多取前 4 个活动，避免 MCP 调用过多。
3. 对相邻活动两两读取 `item.place`。
4. 用已有 `amap_geocode` 把地点转为坐标。
5. 用已有 `amap_route_distance` 估算相邻地点的距离和耗时。
6. 如果耗时超过阈值，追加中文动线风险提醒到 `plan.tips`。
7. 任一地图调用失败时不中断行程生成，只跳过该段，并在必要时追加通用核对提醒。

## 阈值

先用一个固定阈值：相邻活动通勤预计超过 45 分钟，视为动线风险。

这是刻意的最小规则。后续如果城市、同行人群或节奏需要不同阈值，再扩展为按 `pace` / `companions` 调整。

## 数据与接口

复用现有 MCP 工具：

- `amap_geocode`：输入活动地点文本，输出坐标。
- `amap_route_distance`：输入相邻坐标，输出距离和耗时。

不新增 Pydantic 模型字段。提醒继续写入 `TripPlanResponse.tips`，前端现有结果展示可以直接显示。

## 错误处理

- 地点为空：跳过该段。
- geocode 无结果：跳过该段。
- route distance 无结果：跳过该段。
- MCP 抛错：捕获后继续检查下一段。
- 所有失败都不阻塞最终行程返回。

## 测试

新增或扩展现有 `backend/src/app/graph/test_trip_planner.py`：

1. 当相邻活动路线耗时超过 45 分钟时，`tips` 包含动线风险提醒。
2. 当 geocode 或 route 调用失败时，仍返回原计划，不抛异常。
3. 每天超过 4 个活动时，只检查前 4 个活动对应的相邻段，避免调用爆炸。

## 实施边界

最小改动集中在 `backend/src/app/graph/trip_planner.py` 和对应测试。前端只在现有 `tips` 展示文案，不做结构化风险列表。
