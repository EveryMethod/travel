# 中文用户文案本地化设计

## 背景

当前旅行规划页面的大部分表单文案已是中文，但生成的旅行计划内容由后端演示规划器返回，仍包含英文摘要、每日标题、主题、行程描述、旅行提示和免责声明。页面空状态和介绍文案中也带有偏技术演示的表达，例如强调前端、后端和 LangGraph 链路，不适合作为面向中文用户的产品文案。

本次目标是让用户可见内容以中文为主，并将页面语气调整为中文旅行规划产品的表达方式。

## 目标

- 将所有用户可见的旅行计划生成结果改为中文。
- 将首页说明、空状态等面向用户的文案改为自然中文，弱化技术演示表达。
- 保持现有页面结构、接口结构和前后端通信契约不变。
- 不引入 i18n 或多语言框架；当前产品以中文用户为主要对象。

## 非目标

- 不重构页面布局或视觉风格。
- 不修改 API 路由、请求字段、响应字段或枚举值。
- 不接入真实大模型生成能力；后端仍是本地 LangGraph 演示规划器。
- 不新增多语言切换能力。

## 推荐方案

采用轻量直接中文化方案：在现有前端页面和后端演示规划器中替换用户可见文案，不改变数据结构。

### 前端范围

文件：`front/src/views/HomeView.vue`

需要调整：

- 首页主说明：从技术链路说明改为用户价值说明，例如强调“填写偏好，生成可继续调整的中文行程草案”。
- 空状态说明：从“验证前端、后端和 LangGraph 链路”改为引导用户提交表单生成行程。
- 已有中文表单、按钮、错误提示保留；如发现语气不统一，可轻微润色。

### 后端范围

文件：`backend/src/app/graph/trip_planner.py`

需要调整：

- `_STYLE_THEMES`：将旅行风格对应的主题、锚点地点改为中文。
- `_build_demo_plan()`：生成中文摘要、旅行提示、免责声明。
- `_build_day_plan()`：生成中文每日标题、主题、上午/下午/晚上安排、备注。
- `_budget_tip()`：返回中文预算建议。

保留：

- `trip_id` 格式继续使用 `demo-<slug>-<days>`，因为它是技术标识，不作为主要用户文案展示。
- 请求和响应字段名保持英文，避免破坏前后端类型契约。
- `BudgetLevel`、`TravelStyle` 枚举值保持英文，前端继续通过中文 label 展示。

## 数据流

1. 用户在前端填写中文旅行偏好。
2. 前端通过 `planTrip()` POST 到 `/api/trips/plan`。
3. 后端 `plan_trip_with_graph()` 调用本地 LangGraph 演示规划器。
4. 后端返回结构不变，但文本内容为中文。
5. 前端按现有模板展示中文摘要、每日安排、提示和免责声明。

## 错误处理

- 现有前端错误提示已经是中文，保持当前行为。
- 后端异常仍由 API router 返回通用错误；如错误内容直接展示给用户，应保持中文或由前端统一转换为中文。
- 本次不改变错误状态码或异常结构。

## 验证计划

- 启动后端和前端服务。
- 通过前端页面点击“生成行程”，确认页面展示的摘要、每日安排、提示、免责声明均为中文。
- 通过 `/api/trips/plan` 直接请求一次，确认后端响应中的用户可见字段为中文。
- 探测至少一个相邻路径，例如提交不同旅行风格或预算，确认返回的主题和预算建议仍为中文。

## 实施约束

- 遵循现有代码风格，不增加不必要抽象。
- 只修改与中文用户文案相关的最小文件集合。
- 不引入新依赖。
